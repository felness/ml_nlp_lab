# Отчёт исследования: LLM Credit Scoring + Classical ML

## 1. Описание датасета

**UCI Adult (Census Income) — ID 2**

- **Источник:** UC Irvine Machine Learning Repository
- **Объём:** 48 842 записей, 14 признаков
- **Задача:** предсказать, превышает ли годовой доход $50K
- **Применение к кредитованию:** доход >$50K используется как прокси кредитоспособности — признак способности обслуживать долг

**Распределение классов:** 76% — доход ≤$50K (отказ), 24% — доход >$50K (одобрение). Датасет несбалансирован.

## 2. Архитектура прототипа

```
Текстовое описание клиента
        │
        ▼
   FastAPI /analyze
        │
        ▼
   MCP Client ──► Ollama (Qwen2.5:0.5B)
        │              │
        │         tool_calls
        │              │
        ▼              ▼
   MCP Server с инструментами:
   ├── calculate_credit_score       (детерминированная формула)
   ├── assess_risk_by_metadata      (взвешенные правила)
   └── predict_creditworthiness_ml  (GradientBoosting, UCI Adult)
        │
        ▼
   Финальное решение: APPROVED / REJECTED + обоснование на русском
```

**Стек:** Docker · FastAPI · Ollama · Qwen2.5:0.5B · FastMCP · scikit-learn

Ключевая идея — разделить ответственность:  
LLM работает с текстом и логикой, ML — с числами и предсказаниями.


---
## 3. Классические ML-модели

Модели обучены на 80% данных, тест — 20%.  
Препроцессинг: StandardScaler + OneHotEncoder.

### Logistic Regression

Показала стабильный, но базовый результат:

- Accuracy: 0.85  
- F1: 0.66  

Основная проблема — слабый recall по классу “одобрение” (модель осторожна и часто отказывает).

---

### Gradient Boosting (финальный выбор)

- Accuracy: **0.88**  
- F1: **0.72**  

Даёт лучший баланс precision/recall и заметно лучше ловит класс “>50K”.  
Ожидаемо выигрывает за счёт нелинейности и работы с табличными данными.

---

### Вывод по ML

GradientBoosting — наиболее адекватный baseline для задачи.  
Это типичный случай, где классические модели уверенно обгоняют всё “сложное”, если данные табличные.

---

## 4. Поведение LLM (Qwen2.5:0.5B)

Проверка проводилась на 20 примерах из теста.  
Фичи переводились в текст и отправлялись в `/analyze`.

Результат:

- **95% ответов — UNKNOWN**
- Accuracy: 0.75  
- F1: 0.0  

### Что это значит на практике

Модель фактически **отказывает всем**.

За счёт дисбаланса датасета это даёт “нормальную” accuracy,  
но полностью проваливает задачу по классу APPROVED.

То есть:

> модель не предсказывает — она избегает решения

---

## 5. Выводы

### 1. LLM без дообучения — не работает как скоринг

Qwen2.5:0.5B:

- понимает задачу  
- генерирует связный текст  
- но **не соблюдает формат и не даёт стабильных решений**

F1 = 0.0 — это не “плохо”, это фактически отсутствие модели как классификатора.

---

### 2. Классический ML остаётся сильным baseline

GradientBoosting:

- стабилен  
- интерпретируем  
- даёт адекватные метрики  

Для табличных данных это всё ещё лучший инструмент “из коробки”.

---

### 3. LLM + инструменты — рабочая архитектура

Правильный подход — не пытаться заменить ML LLM’ом, а использовать его как оркестратор:

- LLM → парсинг и логика  
- инструменты → вычисления  

Так система остаётся управляемой и предсказуемой.

---

### 4. Ограничения текущего решения

- Adult ≠ реальные кредитные данные  
- нет истории платежей  
- маленькая LLM плохо справляется с tool-calling  

То есть это PoC, а не production-ready система.

---

### 5. Что улучшать дальше

- модель ≥7B параметров или API (GPT / Claude / etc.)  
- реальные кредитные датасеты  
- SHAP для объяснимости  
- калибровка вероятностей  
- ансамбли моделей  

---

## Итог

LLM в текущем виде — плохой скорер, но хороший интерфейс.  
ML — хороший скорер, но плохой интерфейс.  

Комбинация даёт рабочую систему.
